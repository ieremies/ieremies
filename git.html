Cuidado!

Não confunda git com as diversas interfaces gráficas que existem por aí. O VSCode tem a sua forma de mascarar git com a interface dele, mas, no fundo, ele está rodando comandos por você. Eventualmente, talvez uma interface gráfica seja o que você precisa, mas, primeiro, você precisa entender o que essas coisas são e o que elas fazem, antes de escondê-las por debaixo dos panos. Afinal de contas, você está se formando para ser capaz de fazer coisas como essas!

Além disso, lembre-se que GitLab e GitHub são apenas serviços de hospedar seus repositórios! Por mais dominantes que eles são, não há garantias que amanhã eles não sumirão… Já git, não. Git é eterno, o primeiro e o último, o constante!

Instalação

Antes de mais nada, você já sabe, mas não custa lembrar: não use Windows.

Dito isso, instalar valgrind no Linux é fácil: utilizando o seu gerenciador de pacote favorito (apt, pacman, yum, dnf, snap…) instale o pacote git. Muitas distribuições de Linux já terão o git instalado, mas não custa nada confirmar. 

sudo apt install git

Para aqueles no MacOS, pode ser feito algo similar utilizando o brew.

Depois, só configurar o git: 

git config --global user.email "XXXXXX@dac.unicamp.br"
git config --global user.name "Fulana da Silva"
git config credential.helper store # salva a senha na próxima vez que logar

Erros comuns

Não consigo dar push

Talvez, ao dar git push você se depare com este erro: 

warning: redirecting to https://gitlab.ic.unicamp.br/mc202-2018/ra217938.git/
To https://gitlab.ic.unicamp.br/mc202-2018/ra217938
    ! [rejected]        main -> main (fetch first)
error: failed to push some refs to 'https://gitlab.ic.unicamp.br/mc202-2018/ra217938'
hint: Updates were rejected because the remote contains work that you do
hint: not have locally. This is usually caused by another repository pushing
hint: to the same ref. You may want to first integrate the remote changes
hint: (e.g., 'git pull ...') before pushing again.
hint: See the 'Note about fast-forwards' in 'git push --help' for details.

Isso acontece pois alguém modificou o repositório no GitLab e você ainda não tem as alterações feitas. Assim, você deve primeiro fazer git pull --rebase para depois enviar. O --rebase permite que as modificações feitas no servidor do GitLab sejam registradas no seu computador, mas fique tranquilo, o que você fez não corre risco de ser perdido.

Faltam 30min e eu preciso entregar!

De dentro da pasta que suas tarefas existem:

  1. git pull --rebase para pegar a nova tarefa.
  2. git commit -am "mensagem relevante" para "comitar" seu trabalho.
  3. git push para enviar sua tarefa (caso dê erro, volte ao passo 1)
